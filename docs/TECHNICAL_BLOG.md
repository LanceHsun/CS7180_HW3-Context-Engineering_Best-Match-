# Moving Beyond Search Anxiety: How My Teammate and I Built BestMatch

**By Zixin Lin (Frontend Lead) & Yun Feng (Backend & AI Lead)**

Job searching is often more tiring than it should be. In my own experience, I noticed that most job seekers face "Search Anxiety." We jump between different platforms and rewrite our resumes constantly to match job descriptions. Yet, we rarely get efficient feedback.

This is why we built **BestMatch**. I am **Zixin Lin (Frontend Lead)**. Along with my teammate **Yun Feng (Backend & AI Lead)**, we wanted to change the job search logic. You should not have to rewrite your resume for every job. Instead, our system uses your "Master Resume" to find the best-fitting roles from thousands of listings.

In this blog post, we will share how we used Next.js 15, Tailwind CSS v4, and Gemini AI to turn a messy PDF into a precise match score.

---

## Frontend Interaction: Frictionless and Controlled

I believe "Experience is the Product." Yun Feng and I used technology to remove user friction. As the Frontend Lead, I wanted the entry point to be completely frictionless.

### 1. Frictionless Onboarding

New users just drag and drop their resume into the browser. We use **Supabase Magic Links** for authentication. There is no need for traditional passwords.
Technically, we built an "Anonymous Storage" mechanism. When a user drops a file, the frontend creates a temporary ID. The data goes into the `pending_profiles` table. After the user verifies their email, we link the data to their real UID. To mask the AI processing time, I designed a **`ScanLine`** animation using Framer Motion. This provides visual feedback while the system extracts the professional "DNA" from the PDF.

### 2. Mapping Your Professional "DNA"

On the Dashboard, you see a visual skill map. These labels are rendered from the structured JSON generated by Gemini 1.5 Pro. AI is not always perfect, so I made the **"Target Role"** card fully editable. I used the `useActionState` hook for form submission and `useOptimistic` for instant UI updates. If you change your role, the system syncs to the database immediately.

### 3. Real-time Feedback and Preferences

Users can click the **"Search Now"** button to trigger an instant match cycle.

- **Custom ScoreBadges**: These badges show the match level with different colors:
  - **Green (>90%)**: Perfect Fit.
  - **Blue (80-90%)**: Strongly Recommended.
  - **Orange (70-80%)**: Potential Interest.
- **Personalized Preferences**: Users can add or remove location filters or change email frequency. I used **Zod** to validate these inputs, ensuring the matching engine always gets clean data.

## System Architecture & Technical Foundation

Thank you, Zixin. As the Backend Lead, I focused on building a resilient engine capable of handling messy real-world data at scale. Our stack is **Next.js 15** API routes on Vercel, **Supabase** (PostgreSQL + Auth), **Google Gemini 1.5 Pro** (AI scoring), **Adzuna API** (job data), and **SendGrid** (match emails).

We chose Next.js 15 specifically for its seamless integration of Server Actions and the new React 19 hooks. This allowed us to keep all logic in TypeScript, sharing validation schemas between the frontend and backend. Using a serverless architecture on Vercel means we don't have to manage long-running servers, and the system scales automatically as more users upload resumes.

### 1. Database Architecture: Our Modeling Logic

The database design supports the entire business loop. We designed these core models to ensure data integrity and query performance:

| Table Name    | Purpose                                      | Relationship                       |
| ------------- | -------------------------------------------- | ---------------------------------- |
| `profiles`    | Stores candidate professional DNA            | Linked to `auth.users` (1:1)       |
| `preferences` | Stores location filters and digest frequency | Independent from Profile (1:1)     |
| `matches`     | Stores high-score jobs validated by AI       | Linked to User (1:N)               |
| `match_runs`  | Audit logs for batch matching cycles         | Records status, counts, and errors |

We utilized **PostgreSQL Foreign Keys** with `ON DELETE CASCADE` to ensure that if a user deletes their account, all related profile, preference, and match data is purged instantly, complying with data privacy best practices.

### 2. Engineering for Security and Scale

- **Row Level Security (RLS)**: Security is not just a middleware layer here; it’s built into the database. Every table has RLS enabled. Every SQL query filtered by the database engine based on the `auth.uid()` from the user's JWT. This ensures zero data leakage between users.
- **The "Pending Profile" Pattern**: This was our answer to the onboarding friction problem. New users can upload resumes before they even have an account. We use a write-only RLS policy on the `pending_profiles` table. This acts as a "one-way drop box" where the frontend can drop resume data but cannot read it back until the user authenticates.
- **Cleanup Automation**: To prevent the database from bloating with abandoned resumes, a `pg_cron` job runs every hour. It deletes any entries in the pending table that are older than 24 hours, keeping our infrastructure lean and efficient.

## The AI Matching Engine & Job Ingestion

Turning a conversational resume into a mathematical match score requires a two-stage pipeline.

### 1. Multi-Stage AI Pipeline

When a resume is uploaded, we don't just "read" it. We use Gemini 1.5 Pro to perform **Persona Extraction**. We use strict Zod schemas to force the AI to output valid JSON containing target roles, standardized skill sets, and experience levels.

In the matching phase, the `scoreJobMatch()` function acts as a digital recruiter. We use a highly specific **Recruiter Persona Prompt** that instructs the AI to look for "must-have" vs "nice-to-have" skills. The AI provides a score from 0 to 100. We set a strict gate: only matches scoring ≥ 70% are saved and sent to the user.

### 2. Resilient AI Integration

AI services can be unpredictable. To ensure production reliability, we implemented a **multi-model fallback** in `generateWithFallback()`. This is a triple-nested retry strategy:

- **Key Rotation**: We cycle through multiple API keys to avoid 429 rate limits.
- **Model Cascade**: If the premium Gemini 1.5 Pro model is slow or unavailable, we cascade down through 9 different model variants (like Pro 002, Flash, etc.).
- **Exponential Backoff**: We retry failed calls with increasing delay (up to 3 times) to handle temporary network blips.

### 3. Smart Job Ingestion

Live job listings from Adzuna are messy. We hardened our `lib/jobFetcher.ts` with:

- **In-Memory Cache**: A 1-hour TTL keyed by `role:location` ensures we don't hit external APIs (and their rate limits) for the same search twice.
- **Schema Normalization**: Every job is parsed into a `NormalizedJob` Zod schema. If a listing is missing a title or an apply URL, it is silently skipped rather than crashing the matching run.

## Cron Jobs, Automation, and Idempotency

Automation is the soul of BestMatch. Our system runs a daily "match and notify" cycle at 6:00 AM UTC.

### 1. Unified Execution Path

We spent a lot of time ensuring that clicking the "Search Now" button produces the same result as the morning Cron job. We refactored the logic so both the manual trigger and the automated Vercel Cron call the same internal `runMatchingPipeline()` function. This prevents "logic drift" and ensures that manual searches are correctly logged in dashboard analytics.

### 2. Dual-Layer Idempotency

Cron jobs can sometimes fire twice due to infrastructure retries. To protect our users from duplicate emails, we implemented two layers of protection:

- **Run Level**: The system checks the `match_runs` table. If there is already a "completed" run for that user on that calendar day, the second trigger is ignored.
- **Email Level**: Before SendGrid dispatches an email, it does a final check to ensure no email has been sent for that specific `match_run_id`.

## Takeaways & Future Work

Looking back at our two sprints, three lessons stand out:

1. **Validation is Key**: Using Zod to validate environment variables and API payloads at runtime saved us from hours of debugging deployment errors.
2. **Resilience is a Choice**: Our multi-model fallback showed that you can build reliable apps on top of unpredictable AI services if you design for failure.
3. **POCs Matter**: We initially lost a day to a bugged PDF library. Testing core dependencies in a small Proof of Concept (POC) before full integration is mandatory.

In the future, we plan to expand job sources beyond Adzuna and add a "feedback loop" where users can thumbs up/down matches to fine-tune our Gemini scoring prompts.

BestMatch proved that a small team can build a production-grade AI application by combining a robust full-stack architecture with a disciplined, AI-centric workflow.
